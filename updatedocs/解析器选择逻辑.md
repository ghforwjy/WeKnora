# 解析器选择逻辑详细说明

## 1. 前端配置流程

### 1.2 解析器配置界面
- **位置**：知识库设置 → 解析引擎
- **功能**：为不同文件类型选择解析引擎
- **支持的解析引擎**：
  - 内置 — DocReader 内置解析引擎
  - markitdown — MarkItDown 解析引擎（微软 MarkItDown 库）
  - docling — Docling 解析引擎（转换为 Markdown 格式）
  - MinerU — MinerU 自建服务
  - MinerU Cloud — MinerU 云服务

### 1.3 配置存储
- **前端**：通过 `parser_engine_rules` 数组存储配置
- **后端**：存储在知识库的 `chunking_config` 字段中
- **数据结构**：
  ```json
  {
    "file_types": ["docx", "doc"],
    "engine": "builtin"
  }
  ```

## 2. 后端解析器选择流程

### 2.1 知识库解析器规则解析
- **文件**：`internal/types/knowledgebase.go`
- **方法**：`ResolveParserEngine(fileType string)`
- **逻辑**：遍历 `parser_engine_rules`，根据文件类型匹配对应的解析引擎

### 2.2 解析器服务选择
- **文件**：`internal/application/service/knowledge.go`
- **方法**：`resolveDocReader(engine, fileType string, isURL bool, overrides map[string]string)`
- **逻辑**：
  1. 如果指定了 `builtin` 引擎，使用 `docreader` 服务
  2. 如果指定了 `mineru` 引擎，使用 MinerU 服务
  3. 如果指定了 `mineru_cloud` 引擎，使用 MinerU Cloud 服务
  4. 如果指定了 `simple` 引擎，使用 Go 原生简单格式解析器
  5. 未指定引擎时，简单格式使用 Go 原生解析器，其他格式使用 `docreader` 服务

### 2.3 解析请求构建
- **文件**：`internal/application/service/knowledge.go`
- **逻辑**：构建 `ReadRequest`，包含 `ParserEngine` 和 `ParserEngineOverrides` 参数

## 3. DocReader 服务解析器选择

### 3.1 请求处理
- **文件**：`docreader/main.py`
- **方法**：`Read(request: ReadRequest, context)`
- **逻辑**：从请求中提取 `parser_engine` 和 `parser_engine_overrides` 参数

### 3.2 解析器选择
- **文件**：`docreader/parser/parser.py`
- **方法**：`parse_file(file_name, file_type, content, parser_engine=None, engine_overrides=None)`
- **逻辑**：调用 `registry.get_parser_class(engine, file_type)` 获取解析器类

### 3.3 解析器注册表
- **文件**：`docreader/parser/registry.py`
- **方法**：`get_parser_class(engine, file_type)`
- **逻辑**：
  1. 首先尝试使用指定的引擎
  2. 如果指定引擎不支持该文件类型，回退到内置引擎
  3. 内置引擎映射：
     - `docx` → `DocxParser`
     - `doc` → `DocParser`（转换为 DOCX 后使用 `DocxParser`）
     - `pdf` → `PDFParser`
     - `md`/`markdown` → `MarkdownParser`
     - `xlsx`/`xls` → `ExcelParser`
     - 图片类型 → `ImageParser`

## 4. 解析器执行流程

### 4.1 内置解析引擎
- **DocxParser**：直接解析 docx 文件，保持原始格式
- **DocParser**：解析 doc 文件，执行顺序：
  1. 首先尝试使用 antiword 提取文本（保留原始编号格式）
  2. 如果 antiword 提取失败，尝试将 DOC 转换为 DOCX 后使用 DocxParser 解析
  3. 注释掉了使用 textract 的选项（存在 SSRF 漏洞）
- **PDFParser**：解析 PDF 文件
- **MarkdownParser**：解析 Markdown 文件
- **ExcelParser**：解析 Excel 文件
- **ImageParser**：处理图片文件

### 4.2 MarkItDown 解析引擎
- **MarkitdownParser**：使用微软 MarkItDown 库解析多种文件类型

### 4.3 Docling 解析引擎
- **DoclingParser**：使用 IBM Docling 库解析多种文件类型，转换为 Markdown 格式
- **特点**：
  - 保留文档结构和格式
  - 支持多种文件格式（DOC、DOCX、PDF、PPTX、XLSX 等）
  - 生成的 Markdown 格式更有利于后续的切片处理
  - 提供更准确的布局理解和内容提取

### 4.4 MinerU 解析引擎
- **MinerUReader**：使用 MinerU 自建服务解析文件
- **MinerUCloudReader**：使用 MinerU 云服务解析文件

## 5. 解析结果处理

### 5.1 解析结果结构
- **内容**：解析后的 Markdown 内容
- **图片**：提取的图片引用
- **错误**：解析错误信息

### 5.2 结果返回
- DocReader 服务将解析结果返回给后端
- 后端将解析结果存储到知识库

### 6.1 前端配置示例

```json
    {

      "engine": "markitdown"
    },
    {
      "file_types": ["md", "markdown"],
      "engine": "builtin"
    }
  ]

```

### 6.2 解析器选择示例
- **.docx 文件**：使用内置引擎 → `DocxParser`
- **.pdf 文件**：使用 markitdown 引擎 → `MarkitdownParser`
- **.md 文件**：使用内置引擎 → `MarkdownParser`

## 7. 注意事项

### 7.1 解析器可用性
- 部分解析引擎需要配置相关服务（如 MinerU）
- 解析引擎不可用时会回退到内置引擎

### 7.2 解析效果
- 不同解析引擎的解析效果可能不同
- 内置引擎保持原始格式，MarkItDown 引擎可能改变格式

### 7.3 性能考虑
- 不同解析引擎的性能和处理时间可能不同
- 大型文件建议使用性能较好的解析引擎

## 8. 代码修改说明

### 8.1 关键修改

#### 8.1.1 registry.py - 添加 DocxParser 导入
- **文件**：`docreader/parser/registry.py`
- **修改**：添加 `from docreader.parser.docx_parser import DocxParser` 导入语句
- **原因**：确保 `DocxParser` 类可在注册表中使用

#### 8.1.2 docx2_parser.py - 修改解析器顺序
- **文件**：`docreader/parser/docx2_parser.py`
- **修改**：将 `_parser_cls` 从 `(MarkitdownParser, DocxParser)` 改为 `(DocxParser,)`
- **原因**：确保 `Docx2Parser` 只使用 `DocxParser`，避免优先使用 `MarkitdownParser` 导致的格式变更

#### 8.1.3 doc_parser.py - 修改 DOC 解析逻辑
- **文件**：`docreader/parser/doc_parser.py`
- **修改**：修改 `_parse_with_docx` 方法，直接使用 `DocxParser` 而不是通过 `super(Docx2Parser, self).parse_into_text(docx_content)`
- **原因**：确保 DOC 文件转换为 DOCX 后直接使用 `DocxParser` 解析，避免可能的解析器链回退到 `MarkitdownParser`

#### 8.1.4 doc_parser.py - 修改 antiword 执行方式
- **文件**：`docreader/parser/doc_parser.py`
- **修改**：修改 `_parse_with_antiword` 方法，直接使用 `antiword` 命令，而不是通过 `_try_find_antiword()` 方法
- **原因**：确保 DOC 文件解析时保留原始编号格式（如"第一条"、"第二条"），避免格式丢失

#### 8.1.5 添加 Docling 解析引擎
- **文件**：`docreader/parser/docling_parser.py`
- **修改**：创建新的 `DoclingParser` 类，使用 IBM Docling 库将文档转换为 Markdown 格式
- **原因**：提供更优的 Markdown 转换能力，有利于后续的切片处理

#### 8.1.6 registry.py - 注册 Docling 解析引擎
- **文件**：`docreader/parser/registry.py`
- **修改**：添加 `DoclingParser` 导入并注册为新的解析引擎
- **原因**：使 Docling 解析引擎可在前端配置中选择

#### 8.1.7 添加 Docling 依赖
- **文件**：`docreader/pyproject.toml`
- **修改**：在 dependencies 列表中添加 `docling>=2.80.0`
- **原因**：确保构建镜像时自动安装 Docling 依赖

### 8.2 通过 docker-compose 安装 Docling 依赖

#### 8.2.1 依赖安装流程
- **构建镜像**：当执行 `docker-compose build docreader` 时，Docker 会：
  1. 读取 `docreader/pyproject.toml` 文件
  2. 执行 `python -m uv sync --locked --no-dev` 命令
  3. 自动安装包括 docling 在内的所有依赖

#### 8.2.2 部署验证
在新服务器上部署时，执行以下命令：
```bash
# 复制项目到新服务器
scp -r /path/to/weknora user@new-server:/path/to/weknora

# 进入项目目录
cd /path/to/weknora

# 启动容器
docker-compose up -d

# 验证 docling 是否安装成功
docker exec -it WeKnora-docreader pip list | grep docling
```

#### 8.2.3 优势
- **可迁移性**：所有配置都在项目文件中，无需在每个服务器上手动安装依赖
- **一致性**：使用 Docker 确保在不同环境中依赖版本一致
- **自动化**：构建过程自动安装依赖，无需手动干预
- **灵活性**：通过挂载本地目录，便于开发和调试

```python

```

#### doc_parser.py
```python
def _parse_with_docx(self, temp_file_path: str) -> Document:
    logger.info("Multimodal enabled, attempting to extract images from DOC")

    docx_content = self._try_convert_doc_to_docx(temp_file_path)
    if not docx_content:
        raise RuntimeError("Failed to convert DOC to DOCX")

    logger.info("Successfully converted DOC to DOCX, using DocxParser")
    # Use DocxParser directly to parse the converted docx
    from docreader.parser.docx_parser import DocxParser
    docx_parser = DocxParser(file_name=self.file_name, file_type='docx')
    document = docx_parser.parse_into_text(docx_content)
    logger.info(f"Extracted {len(document.content)} characters using DocxParser")
    return document

def _parse_with_antiword(self, temp_file_path: str) -> Document:
    logger.info("Attempting to parse DOC file with antiword")

    # Use antiword command directly
    cmd = ["antiword", temp_file_path]
    logger.info(f"Executing antiword command: {' '.join(cmd)}")
    logger.info(f"Temp file path: {temp_file_path}")
    logger.info(f"Temp file exists: {os.path.exists(temp_file_path)}")
    if os.path.exists(temp_file_path):
        logger.info(f"Temp file size: {os.path.getsize(temp_file_path)} bytes")

    # Execute antiword command
    import subprocess
    try:
        # Execute antiword command
        result = subprocess.run(cmd, capture_output=True, timeout=60)
        if result.returncode != 0:
            raise RuntimeError(
                f"antiword extraction failed: {result.stderr.decode('utf-8', errors='ignore')}"
            )
        
        # Try different encodings to find the correct one
        encodings = ['utf-8', 'gbk', 'gb2312', 'latin1']
        text = None
        
        for encoding in encodings:
            try:
                text = result.stdout.decode(encoding)
                logger.info(f"Successfully decoded with {encoding}")
                break
            except Exception as e:
                logger.warning(f"Failed to decode with {encoding}: {e}")
                continue
        
        if not text:
            raise RuntimeError("Failed to decode antiword output")
        
        logger.info(f"Successfully extracted {len(text)} characters using antiword")
        
        # Check if numbering is present
        if '第一条' in text or '第二条' in text:
            logger.info("Found numbering format in extracted text")
        else:
            logger.warning("No numbering format found in extracted text")
        
        return Document(content=text)
    except Exception as e:
        logger.error(f"Direct antiword execution failed: {e}")
        import traceback
        traceback.print_exc()
        raise
```

## 9. 故障排除

### 9.1 解析器不生效
- 检查前端配置是否正确保存
- 检查知识库配置是否更新
- 检查 DocReader 服务是否正常运行

### 9.2 解析结果不符合预期
- 尝试更换解析引擎
- 检查文件格式是否支持
- 查看 DocReader 服务日志

## 10. 总结

解析器选择流程从前端配置开始，经过后端处理，最终路由到实际的解析器执行解析操作。通过合理配置解析引擎，可以获得最佳的解析效果。

- **前端**：为不同文件类型选择合适的解析引擎
- **后端**：根据配置选择对应的解析器服务
- **DocReader**：根据引擎和文件类型选择具体的解析器类
- **解析器**：执行实际的文件解析操作

此流程确保了前端配置能够正确影响解析器选择，从而获得期望的解析效果。
