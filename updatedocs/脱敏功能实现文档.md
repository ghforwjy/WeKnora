# 脱敏功能实现文档

## 1. 功能介绍

脱敏功能是 WeKnora 系统中的一个安全特性，用于在文档分块前对敏感信息进行替换，保护用户隐私和敏感数据。该功能允许用户配置敏感词和对应的替换词，系统会在解析文档后、分块前自动对敏感词进行替换处理。

### 主要功能点：
- 支持启用/禁用脱敏功能
- 支持配置多个敏感词和对应的替换词
- 支持在文档分块前自动应用脱敏处理
- 支持父子分块模式下的脱敏处理

## 2. 实现逻辑

脱敏功能的实现分为前端配置和后端处理两个部分：

### 2.1 前端配置
1. 在知识库分块设置界面添加脱敏功能开关
2. 提供敏感词和替换词的输入界面
3. 将配置保存到知识库的 ChunkingConfig 中

### 2.2 后端处理
1. 在文档解析完成后，分块前应用脱敏处理
2. 对所有文本内容进行敏感词替换
3. 支持普通分块和父子分块两种模式的脱敏

### 8.2 命名不一致问题

**问题描述**：数据库存储层使用蛇形命名 `sensitive_config`，而API层使用驼峰命名 `sensitiveConfig`。

**影响**：
- 前端与后端API通信正常（都使用驼峰 `sensitiveConfig`）
- 数据库存储时会进行JSON序列化，使用蛇形 `sensitive_config`
- 数据流转：前端(驼峰) → 后端API(驼峰) → 数据库(蛇形) → 后端API(驼峰) → 前端(驼峰)

**结论**：当前实现可以正常工作，因为后端在API层和数据库层做了正确的JSON映射转换。

## 4. 使用说明

### 4.1 配置示例
**敏感词**：密码,身份证,手机号
**替换为**：***,****,*****

### 4.2 处理流程
1. 用户上传文档
2. 系统解析文档内容
3. 应用脱敏处理，将敏感词替换为指定的替换词
4. 对处理后的文本进行分块
5. 生成向量嵌入并存储

### 4.3 注意事项
- 敏感词和替换词的数量必须一致
- 敏感词匹配是精确匹配，区分大小写
- 脱敏处理会在分块前应用，确保所有分块内容都经过脱敏处理
- 脱敏处理不会修改原始文档，只在分块过程中应用

## 5. 测试验证

1. 配置敏感词和替换词
2. 上传包含敏感信息的测试文档
3. 查看分块结果，确认敏感信息已被替换
4. 进行检索测试，确认检索结果中不包含敏感信息

## 6. 故障排查

### 6.1 脱敏功能不生效
- 检查脱敏功能是否已开启
- 检查敏感词和替换词配置是否正确
- 检查前端是否已重新构建并部署
- 检查后端服务是否正常运行
- 检查后端 API 处理层是否正确处理脱敏配置

### 6.2 脱敏配置保存失败
- 检查后端 `internal/handler/initialization.go` 文件中是否正确添加了脱敏配置处理
- 检查 KBModelConfigRequest 结构体是否包含 SensitiveConfig 字段
- 检查 UpdateKBConfig 函数是否正确更新脱敏配置
- 检查 buildConfigResponse 函数是否正确返回脱敏配置

### 6.3 脱敏效果不符合预期
- 检查敏感词是否正确配置
- 检查敏感词是否区分大小写
- 检查文档中是否包含完全匹配的敏感词

## 10. 问题排查记录

### 10.1 问题描述（已修复）

**问题现象**：脱敏配置保存后，`enabled` 开关能正确显示，但敏感词和替换词输入框为空。

**排查过程**：

#### 步骤1：前端日志调试

在 `KnowledgeBaseEditorModal.vue` 的 `loadKBData` 函数中添加日志：

```javascript
const kb = kbInfo.data
console.log('[DEBUG] kb.chunking_config:', kb.chunking_config)
console.log('[DEBUG] kb.chunking_config?.sensitive_config:', kb.chunking_config?.sensitive_config)
console.log('[DEBUG] kb.chunking_config?.sensitive_config?.enabled:', kb.chunking_config?.sensitive_config?.enabled)
console.log('[DEBUG] kb.chunking_config?.sensitive_config?.replacements:', kb.chunking_config?.sensitive_config?.replacements)
```

**日志输出结果**：
```
[DEBUG] kb.chunking_config: {chunk_size: 512, chunk_overlap: 100, separators: Array(7), ...}
[DEBUG] kb.chunking_config?.sensitive_config: {enabled: true, replacements: {…}}
[DEBUG] kb.chunking_config?.sensitive_config?.enabled: true
[DEBUG] kb.chunking_config?.sensitive_config?.replacements: {aa: 'bb'}
```

**结论**：API 返回数据正确，问题在前端控件显示环节。

#### 步骤2：组件初始化逻辑审查

检查 `KBChunkingSettings.vue` 组件的初始化代码：

```javascript
// 修复前的问题代码
const localSensitiveEnabled = ref(props.config.sensitiveConfig?.enabled ?? false)
const localSensitiveWords = ref('')        // ❌ 初始化为空字符串
const localReplacementWords = ref('')      // ❌ 初始化为空字符串
```

**根本原因**：初始化时直接使用空字符串，没有从 `props.config.sensitiveConfig?.replacements` 读取已保存的数据！

### 10.2 修复方案

修改 `KBChunkingSettings.vue` 组件的初始化逻辑：

```javascript
// 修复后
const localSensitiveEnabled = ref(props.config.sensitiveConfig?.enabled ?? false)
const initialReplacements = props.config.sensitiveConfig?.replacements || {}
const initialSensitiveWords = Object.keys(initialReplacements).join(', ')
const initialReplacementWords = Object.values(initialReplacements).join(', ')
const localSensitiveWords = ref(initialSensitiveWords)
const localReplacementWords = ref(initialReplacementWords)
```

| **影响范围** | 仅影响敏感词和替换词显示，enabled 开关不受影响 |
