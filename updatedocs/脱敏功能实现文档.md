# 脱敏功能实现文档

## 1. 功能介绍

脱敏功能是 WeKnora 系统中的一个安全特性，用于在文档分块前对敏感信息进行替换，保护用户隐私和敏感数据。该功能允许用户配置敏感词和对应的替换词，系统会在解析文档后、分块前自动对敏感词进行替换处理。

### 主要功能点：
- 支持启用/禁用脱敏功能
- 支持配置多个敏感词和对应的替换词
- 支持在文档分块前自动应用脱敏处理
- 支持父子分块模式下的脱敏处理

## 2. 实现逻辑

脱敏功能的实现分为前端配置和后端处理两个部分：

### 2.1 前端配置
1. 在知识库分块设置界面添加脱敏功能开关
2. 提供敏感词和替换词的输入界面
3. 将配置保存到知识库的 ChunkingConfig 中

### 2.2 后端处理
1. 在文档解析完成后，分块前应用脱敏处理
2. 对所有文本内容进行敏感词替换
3. 支持普通分块和父子分块两种模式的脱敏

## 4. 使用说明

### 4.1 配置示例
**敏感词**：密码,身份证,手机号
**替换为**：***,****,*****

### 4.2 处理流程
1. 用户上传文档
2. 系统解析文档内容
3. 应用脱敏处理，将敏感词替换为指定的替换词
4. 对处理后的文本进行分块
5. 生成向量嵌入并存储

### 4.3 注意事项
- 敏感词和替换词的数量必须一致
- 敏感词匹配是精确匹配，区分大小写
- 脱敏处理会在分块前应用，确保所有分块内容都经过脱敏处理
- 脱敏处理不会修改原始文档，只在分块过程中应用

## 5. 测试验证

1. 配置敏感词和替换词
2. 上传包含敏感信息的测试文档
3. 查看分块结果，确认敏感信息已被替换
4. 进行检索测试，确认检索结果中不包含敏感信息

## 6. 故障排查

### 6.1 脱敏功能不生效
- 检查脱敏功能是否已开启
- 检查敏感词和替换词配置是否正确
- 检查前端是否已重新构建并部署
- 检查后端服务是否正常运行
- 检查后端 API 处理层是否正确处理脱敏配置

### 6.2 脱敏配置保存失败
- 检查后端 `internal/handler/initialization.go` 文件中是否正确添加了脱敏配置处理
- 检查 KBModelConfigRequest 结构体是否包含 SensitiveConfig 字段
- 检查 UpdateKBConfig 函数是否正确更新脱敏配置
- 检查 buildConfigResponse 函数是否正确返回脱敏配置

### 6.3 脱敏效果不符合预期
- 检查敏感词是否正确配置
- 检查敏感词是否区分大小写
- 检查文档中是否包含完全匹配的敏感词

### 10.2 问题排查过程

#### 数据流断点定位

```
前端发送正确数据 → API返回成功 → 【后端 UpdateKBConfig 函数中保存脱敏配置的代码从未执行】
```

### 10.3 根本原因

**后端代码修改从未被编译进运行的二进制文件**

#### 容器目录结构

**app服务（WeKnora-app）挂载情况：**

| 本地路径 | 容器路径 | 权限 | 用途 |
|----------|----------|------|------|
| `./` | `/app_src` | 只读`:ro` | 后端源码挂载 |
| `./scripts` | `/app/scripts` | 只读`:ro` | 脚本目录 |
| `./volumes/data_files` | `/data/files` | 可写 | 数据文件存储 |
| `./volumes/docreader-tmp` | `/tmp/docreader` | 只读`:ro` | docreader临时文件 |
| `./config/config.yaml` | `/app/config/config.yaml` | 可写 | 配置文件 |
| `./config/prompt_templates` | `/app/config/prompt_templates` | 只读`:ro` | 提示模板 |
| `./config/builtin_agents.yaml` | `/app/config/builtin_agents.yaml` | 只读`:ro` | 内置代理配置 |
| `./volumes/go-build-cache` | `/root/.cache/go-build` | 可写 | Go编译缓存 |
| `./volumes/pip-cache` | `/root/.cache/pip` | 可写 | pip缓存 |
| `./volumes/uv-cache` | `/root/.cache/uv` | 可写 | uv缓存 |
| `./volumes/duckdb` | `/home/appuser/.duckdb` | 可写 | DuckDB数据 |
| 无映射 | `/app` | 无 | 运行时目录（来自镜像，包含二进制文件WeKnora） |

**关键点：**
- **/app_src**：源码挂载，**只读**，修改后重启恢复原状
- **/app**：二进制文件所在目录，**无本地映射**，无法持久化修改
- 其他缓存和数据目录有本地映射，可持久化

#### 原因1：容器挂载限制
- `/app_src` 是**只读挂载**（`:ro`），无法在容器内直接编译Go代码
- `/app` 目录（二进制文件所在位置）**没有映射到本地文件系统**，无法持久化编译结果

#### 原因2：编译依赖缺失
- 项目依赖 `go-sqlite3`（SQLite数据库的Go语言绑定），使用cgo调用C库
- 编译时报错缺少 `sqlite3.h`：需要安装SQLite开发库
- 容器是精简镜像，默认不包含开发库

#### 原因3：编译结果无法持久化
- 尝试在容器内编译成功后，替换二进制文件并重启容器
- 但容器重启后使用镜像中的原始二进制文件，编译结果丢失
- `docker-compose up` 会重新创建容器，之前的修改会丢失

### 10.4 编译环境配置方案

#### 10.4.1 问题分析

容器内编译后端代码需要解决三个问题：

| 序号 | 问题 | 当前状态 | 需要改为 |
|------|------|----------|----------|
| 1 | 代码可编辑 | `/app_src` 只读挂载 (`:ro`) | 改为可写挂载 |
| 2 | 二进制持久化 | `/app` 无本地映射 | 映射到宿主机目录 |
| 3 | 编译依赖 | 容器缺少 sqlite3 开发库 | 在容器内安装依赖 |

#### 10.4.2 解决方案：临时容器编译跳板法

**核心思想**：在独立于主容器的临时环境中编译代码，编译成功后二进制文件直接写入主容器挂载的本地目录。**关键价值**：当编译出错导致二进制文件不可用时，主容器仍然正常运行，可以继续修复代码、重新编译、反复尝试，而不会陷入"编译失败→容器起不来→无法修复代码"的死循环。

**步骤1：修改 docker-compose.yml**

```yaml
# app 服务添加两个挂载
volumes:
  # 1. 源码改为可写
  - ./:/app_src:rw
  # 2. 二进制目录映射到本地（持久化编译结果）
  - ./volumes/app:/app
```

**步骤2：创建编译跳板容器**

```bash
# 创建临时容器（保留，作为编译跳板）
docker run -d --name WeKnora-app-temp \
  --network weknora_WeKnora-network \
  -v E:/mycode/weknora:/app_src:rw \
  -v E:/mycode/weknora/volumes/app:/app \
  golang:1.24-bookworm sleep 3600
```

**步骤3：在跳板容器中安装依赖并编译**

```bash
# 首次使用需要安装编译依赖（只需执行一次）
docker exec WeKnora-app-temp bash -c 'apt-get update && apt-get install -y libsqlite3-dev'

# 编译 Go 代码
docker exec WeKnora-app-temp bash -c 'cd /app_src/cmd/server && go build -o /app/WeKnora .'
```

**注意**：编译结果直接写入 `/app/WeKnora`（映射到宿主机 `volumes/app`），无需单独拷贝步骤。

**Protobuf 冲突解决**：运行时错误 `proto: file "common.proto" is already registered` 通过在 docker-compose.yml 中添加环境变量解决：

```yaml
environment:
  - GOLANG_PROTOBUF_REGISTRATION_CONFLICT=warn
```

**步骤4：重启主容器生效**

```bash
# 编译完成后重启主容器
docker restart WeKnora-app
```

#### 10.4.3 跳板容器维护

| 操作 | 命令 |
|------|------|
| **保留跳板容器** | 跳板容器稳定运行时应保留，避免每次重新下载依赖 |
| **删除并重建** | 仅在容器异常或需要重置时执行：`docker rm -f WeKnora-app-temp` 后重新创建 |
| **验证源码** | `docker exec WeKnora-app-temp sed -n '310,320p' /app_src/internal/handler/initialization.go` |
| **验证编译结果** | `grep -c "Before saving to database" /app/WeKnora` |
| **清理缓存** | `docker exec WeKnora-app-temp bash -c 'cd /app_src && go clean -cache'` |

**注意**：编译结果直接写入 `/app/WeKnora`（映射到宿主机 `volumes/app`），无需单独拷贝步骤。

#### 10.4.4 关键经验

| 经验 | 说明 |
|------|------|
| **临时容器做跳板** | 编译失败不影响主容器，可以反复重试 |
| **编译结果持久化** | 二进制文件写入 `/app`（映射到宿主机 `volumes/app`），重启后不丢失 |
| **先验证源码** | 用 `sed -n '310,320p' /app_src/internal/handler/initialization.go` 确认代码存在 |
| **清理缓存** | `go clean -cache` 避免缓存导致编译问题 |
| **LDFLAGS 必须设置** | protobuf 冲突需要添加 `protoregistry.conflictPolicy=warn` |
